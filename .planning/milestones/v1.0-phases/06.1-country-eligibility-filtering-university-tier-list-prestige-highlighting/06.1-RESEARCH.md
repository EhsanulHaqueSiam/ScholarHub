# Phase 06.1: Country Eligibility Filtering, University Tier List, Prestige Highlighting - Research

**Researched:** 2026-03-20
**Domain:** Full-stack React SSR (TanStack Start), Convex filtering/search, prestige scoring, country eligibility, neo-brutalism UI
**Confidence:** MEDIUM-HIGH

## Summary

This phase is the largest single phase in the project, combining the entire public directory (Phase 6 absorbed) with country eligibility filtering and a prestige tier system. The core technical challenges are: (1) migrating from TanStack Router SPA to TanStack Start SSR for dynamic meta tags/SEO, (2) building a performant server-side filtering system in Convex with search, compound indexes, and pagination within free-tier limits, (3) implementing a write-time prestige scoring system, and (4) building a rich filtering UI with full URL state management.

The TanStack Start migration is the riskiest item -- it changes the fundamental app architecture from client-side SPA to SSR framework. The project currently uses `TanStackRouterVite` plugin, `src/main.tsx` entry point, and Netlify SPA redirect rules -- all of which must change. The good news is that Netlify is TanStack Start's official deployment partner with a dedicated Vite plugin, and the existing `src/routes/` directory convention aligns with TanStack Start's default.

The Convex filtering challenge is significant due to search index limitations: only one searchField per index, max 16 filterFields, max 1024 documents scanned per search query. For a dataset of 2000-5000 scholarships, this means the search approach must be carefully designed. The recommended pattern is a denormalized `search_text` field combining title + description + eligibility text, with filterFields for the most common equality filters (status, host_country, funding_type, prestige_tier).

**Primary recommendation:** Begin with TanStack Start migration as first wave since it is infrastructure that all other work depends on, then build Convex schema/queries, then UI components, then wire everything together.

### Pattern 3: Write-Time Prestige Scoring with Triggers
**What:** Calculate prestige score and tier automatically whenever scholarship data changes
**When to use:** Use convex-helpers triggers on the scholarships table

```typescript
// web/convex/prestige.ts
// Prestige weight defaults (admin-configurable later)
const DEFAULT_WEIGHTS = {
  funding_level: 0.40,
  provider_reputation: 0.30,
  country_rank: 0.20,
  competitiveness: 0.10,
};

// Score each factor 0-100, then weighted average
function calculatePrestigeScore(scholarship: Doc<"scholarships">): number {
  const fundingScore = scoreFunding(scholarship.funding_type);
  const providerScore = scoreProvider(scholarship.provider_organization);
  const countryScore = scoreCountry(scholarship.host_country);
  const competitivenessScore = scoreCompetitiveness(scholarship);

  return Math.round(
    fundingScore * DEFAULT_WEIGHTS.funding_level +
    providerScore * DEFAULT_WEIGHTS.provider_reputation +
    countryScore * DEFAULT_WEIGHTS.country_rank +
    competitivenessScore * DEFAULT_WEIGHTS.competitiveness
  );
}

function scoreTier(score: number): PrestigeTier {
  if (score >= 75) return "gold";
  if (score >= 50) return "silver";
  if (score >= 25) return "bronze";
  return "unranked";
}
```

### Pattern 5: Convex Server-Side Filtering
**What:** Efficient server-side filtering within Convex free tier limits
**When to use:** For all directory listing queries

The Convex filtering strategy must handle the constraint that search indexes only support equality filters (`.eq`), not array membership or OR logic. The approach:

1. **For text search**: Use the `search_text` denormalized field with `.withSearchIndex()`
2. **For equality filters** (status, funding_type, prestige_tier): Use search index filterFields with `.eq()`
3. **For array membership** (nationality in eligibility_nationalities, degree in degree_levels): Use `.filter()` post-processing
4. **For multi-value filters** (multiple countries, multiple degrees): Client-side OR logic applied via `.filter()`
5. **For sorting**: Separate indexed queries per sort order (use compound indexes)

```typescript
// web/convex/directory.ts
export const listScholarships = query({
  args: {
    paginationOpts: paginationOptsValidator,
    search: v.optional(v.string()),
    status: v.optional(scholarshipStatusValidator),
    hostCountries: v.optional(v.array(v.string())),
    nationalities: v.optional(v.array(v.string())),
    degreeLevels: v.optional(v.array(degreeLevelValidator)),
    fundingType: v.optional(fundingTypeValidator),
    prestigeTier: v.optional(prestigeTierValidator),
    sort: v.optional(v.string()),
  },
  handler: async (ctx, args) => {
    let query;

    if (args.search) {
      // Full-text search path
      query = ctx.db
        .query("scholarships")
        .withSearchIndex("search_scholarships", (q) => {
          let sq = q.search("search_text", args.search!);
          if (args.status) sq = sq.eq("status", args.status);
          if (args.fundingType) sq = sq.eq("funding_type", args.fundingType);
          if (args.prestigeTier) sq = sq.eq("prestige_tier", args.prestigeTier);
          return sq;
        });
    } else {
      // Index-based path (no search text)
      query = ctx.db
        .query("scholarships")
        .withIndex("by_status_deadline", (q) =>
          q.eq("status", args.status ?? "published")
        )
        .order("asc"); // deadline soonest first
    }

    // Post-filter for array membership (not indexable)
    const results = query.filter((q) => {
      const conditions = [];
      // Host country filter (OR logic)
      if (args.hostCountries?.length) {
        conditions.push(
          q.or(...args.hostCountries.map((c) => q.eq(q.field("host_country"), c)))
        );
      }
      // Nationality eligibility (check if user's nationality is in the array)
      // This requires custom filter logic
      // ...
      return conditions.length ? q.and(...conditions) : true;
    });

    return await results.paginate(args.paginationOpts);
  },
});
```

### Pattern 6: Nationality Detection
**What:** Detect user's likely country from browser signals
**When to use:** On first visit, show soft suggestion banner

**Recommendation:** Use `Intl.DateTimeFormat().resolvedOptions().timeZone` as the primary signal. This is the most reliable browser-only approach -- it returns IANA timezone identifiers (e.g., "Asia/Dhaka") that map to countries with high confidence. Supplement with `navigator.language` as a secondary signal.

**Why not IP-based:** Requires a server call or third-party API, adds latency, blocked by VPNs, privacy concerns. The timezone approach works entirely client-side and covers the vast majority of cases.

```typescript
// Timezone -> Country mapping (simplified)
const TIMEZONE_TO_COUNTRY: Record<string, string> = {
  "Asia/Dhaka": "BD",
  "Asia/Kolkata": "IN",
  "Asia/Calcutta": "IN",
  "America/New_York": "US",
  "Europe/London": "GB",
  "Europe/Berlin": "DE",
  // ... ~200 timezone -> country mappings
};

function detectNationality(): string | null {
  const tz = Intl.DateTimeFormat().resolvedOptions().timeZone;
  return TIMEZONE_TO_COUNTRY[tz] ?? null;
}
```

### Anti-Patterns to Avoid

- **Client-side filtering of all scholarships:** With 2000-5000 records, never fetch all and filter client-side. Always use Convex server-side queries.
- **Separate API call for each filter change:** Convex reactive queries auto-update. Don't debounce filter changes with separate fetch calls -- use Convex's subscription model.
- **Multiple search indexes on same table:** Convex allows only one search index per table (practically). Use the denormalized `search_text` field pattern instead.
- **Auto-filtering without user consent:** Never automatically apply the nationality filter based on detection. Always show a suggestion banner and let the user explicitly set it.
- **Calculating prestige at read time:** With 2000-5000 records and potential concurrent users, this would be expensive. Calculate at write time (mutation/trigger) and store the result.

| Problem | Don't Build | Use Instead | Why |

## Common Pitfalls

### Pitfall 1: TanStack Start Migration Breaking Existing Routes
**What goes wrong:** Routes defined with `createFileRoute` and `createRootRoute` may break when switching to `createRootRouteWithContext` with the new HTML shell.
**Why it happens:** TanStack Start expects the root route to render the full HTML document (`<html>`, `<head>`, `<body>`), not just a React component tree.
**How to avoid:** Migrate root route first, verify it renders correctly, then update child routes one at a time. The existing `index.tsx` route should work unchanged since `createFileRoute` is the same API.
**Warning signs:** Blank page, hydration mismatches in console, missing styles.

### Pitfall 2: Convex Search Index Limitations
**What goes wrong:** Attempting to search across title + description + eligibility text with separate fields fails because Convex only allows one `searchField` per search index.
**Why it happens:** Convex search indexes are designed around a single searchable text field with equality filter fields.
**How to avoid:** Create a denormalized `search_text` field that concatenates title + description + eligibility text. Update it via trigger whenever any of those fields change.
**Warning signs:** Search only matching titles, missing results from description/eligibility text.

### Pitfall 3: Convex 1024 Document Scan Limit on Search
**What goes wrong:** Search queries on a table with 5000 documents may not scan all matching results, leading to incomplete results.
**Why it happens:** Convex limits search queries to scanning 1024 documents maximum.
**How to avoid:** Use filterFields aggressively to narrow the scan set before text matching. Always include `status: "published"` as a filterField. For the expected 2000-5000 dataset, this should be manageable if filters are applied at the index level.
**Warning signs:** Users reporting "I know this scholarship exists but search doesn't find it." Monitor by comparing result counts.

### Pitfall 4: Netlify Config Not Updated for SSR
**What goes wrong:** After TanStack Start migration, the app still serves as SPA because Netlify redirects all routes to `index.html`.
**Why it happens:** The existing `netlify.toml` has SPA redirect rules (`/* -> /index.html`) that bypass SSR.
**How to avoid:** Remove the SPA redirect rule, update publish directory to `dist/client`, ensure `@netlify/vite-plugin-tanstack-start` is installed. The Netlify plugin handles server function deployment automatically.
**Warning signs:** SSR meta tags not appearing in page source, OG tags not working in social previews.

### Pitfall 5: Nationality Filter Array Matching in Convex
**What goes wrong:** Checking if a user's nationality is in `eligibility_nationalities` array is not directly indexable in Convex.
**Why it happens:** Convex indexes work on scalar field equality, not array membership.
**How to avoid:** Use `.filter()` for array membership checks after narrowing results with indexed fields. For the "open to all" case, scholarships with null/empty `eligibility_nationalities` should always be included. Structure the query: index narrows by status + other indexed fields, then `.filter()` checks nationality membership.
**Warning signs:** Slow queries, missing "open to all" scholarships, incorrect eligibility filtering.

### Pitfall 6: Prestige Score Infinite Loop with Triggers
**What goes wrong:** If the prestige calculation trigger writes back to the same table, it may trigger itself again.
**Why it happens:** convex-helpers triggers fire on all mutations to the table, including those made by other triggers.
**How to avoid:** Use `ctx.innerDb` for the prestige score write-back to avoid re-triggering. Or implement a guard that checks if prestige_tier actually changed before writing.
**Warning signs:** Convex function timeout errors, unexpectedly high function call counts on free tier.

### Pitfall 7: Flash of Unstyled Content (FOUC) with Dark Mode + SSR
**What goes wrong:** Page flashes in light mode before dark mode class is applied.
**Why it happens:** SSR renders HTML without knowing the user's theme preference (stored in localStorage).
**How to avoid:** Use TanStack Router's `ScriptOnce` in the root route to inject a blocking script that reads localStorage and sets the `dark` class before the page renders. This script runs before React hydration.
**Warning signs:** Visible light-to-dark flash on page load for dark mode users.

### Pitfall 8: Free Tier Function Call Budget
**What goes wrong:** Reactive subscriptions from multiple concurrent users plus search/filter queries exhaust the 1M calls/month free tier.
**Why it happens:** Each Convex reactive subscription is a function call, and every filter/sort change triggers a new query.
**How to avoid:** Use pagination (not infinite scroll) to limit subscriptions. Consider `gcTime` on React Query to control subscription lifetime. Monitor usage in Convex dashboard. For the expected traffic of a new site, 1M calls/month should be sufficient, but be aware of the budget.
**Warning signs:** Approaching 75% of free tier limit in Convex dashboard.

```typescript
// Client-side usage in directory component
import { usePaginatedQuery } from "convex/react";
import { api } from "../../convex/_generated/api";

function ScholarshipResults({ filters }) {
  const { results, status, loadMore } = usePaginatedQuery(
    api.directory.listScholarships,
    {
      status: "published",
      hostCountries: filters.to,
      nationalities: filters.from,
      degreeLevels: filters.degree,
      fundingType: filters.funding,
      prestigeTier: filters.tier,
      search: filters.q,
      sort: filters.sort,
    },
    { initialNumItems: 20 }
  );

  return (
    <>
      <ResultsCount count={results.length} />
      <div className="grid grid-cols-1 md:grid-cols-2 lg:grid-cols-3 gap-4">
        {results.map((scholarship) => (
          <ScholarshipCard key={scholarship._id} scholarship={scholarship} />
        ))}
      </div>
      {status === "CanLoadMore" && (
        <Button onClick={() => loadMore(20)} variant="neutral">
          Load More
        </Button>
      )}
      {status === "LoadingMore" && <SkeletonCards count={3} />}
    </>
  );
}
```

## Open Questions

1. **Convex + TanStack Start SSR hydration with reactive subscriptions**
   - What we know: Convex docs say SSR renders a consistent snapshot, then hydrates to live subscriptions
   - What's unclear: Whether `usePaginatedQuery` works seamlessly with SSR hydration in TanStack Start or requires special handling
   - Recommendation: Test early in the TanStack Start migration wave. If issues arise, fall back to client-only rendering for the directory page (SSR the shell, client-render the data).

2. **Search index scan limit with growing dataset**
   - What we know: Convex limits to 1024 documents scanned per search query. Current dataset is 2000-5000.
   - What's unclear: Whether aggressive filterField usage keeps scan count under 1024 for common queries
   - Recommendation: Monitor search result completeness. If the dataset approaches 5000 published scholarships, consider partitioning queries or using compound indexes instead of search for non-text-search filter combinations.

3. **Convex filter() performance on array fields**
   - What we know: `.filter()` is applied after index scan, checking each document
   - What's unclear: Performance characteristics when filtering `eligibility_nationalities` array membership on 2000+ documents
   - Recommendation: Use indexed fields to narrow first, then `.filter()` for array checks. Monitor query latency. If slow, consider denormalizing nationality eligibility into a separate indexed field structure.

4. **TanStack Start stability**
   - What we know: TanStack Start is described as "beta" in some docs but has official Netlify partnership
   - What's unclear: Whether there will be breaking changes before stable release
   - Recommendation: Pin versions carefully. The core routing API (`createFileRoute`, `createRootRouteWithContext`) is stable since it's shared with TanStack Router.

## Discretion Recommendations

### Auto-detect nationality approach
**Recommendation: Timezone-based via `Intl.DateTimeFormat().resolvedOptions().timeZone`**
- Most reliable client-side approach, no network call needed
- IANA timezone strings map to countries with high confidence
- Supplement with `navigator.language` as fallback hint
- Covers ~200 timezones to ~240 countries
- Does not require any third-party API or privacy consent

### Filter panel layout
**Recommendation: Sidebar on desktop, bottom sheet on mobile**
- Desktop: left sidebar (240-280px) with sticky positioning, always visible alongside results
- Mobile: "Filters" button at top triggers a bottom sheet (Radix Dialog as sheet)
- Smart eligibility bar stays above results on both desktop and mobile (not in sidebar)
- This maximizes horizontal space for cards on desktop while keeping filters accessible

### Pagination approach
**Recommendation: "Load More" button**
- Best for Convex free tier -- user controls when to fetch more
- Simpler than infinite scroll (no intersection observer, no accidental over-fetching)
- Better than numbered pages for a reactive list that updates in real-time
- Use `usePaginatedQuery` with `initialNumItems: 20`, load 20 more per click
- Show total count in header so users know the full dataset size

### Mobile featured row
**Recommendation: Horizontal scroll with scroll snap**
- Horizontal scroll fits the "featured" mental model (curated picks, not a full list)
- Use CSS scroll-snap for polished feel
- Show left/right peek to indicate scrollability
- Simpler than vertical stack which adds to the already-long page

### URL slug strategy
**Recommendation: `{provider}-{title-slug}` format**
- Example: `fulbright-foreign-student-program`, `daad-research-grants`
- Generate at write time, store in existing `slug` field on scholarships table
- Collision handling: append numeric suffix if slug exists
- Matches the existing `by_slug` index on the scholarships table
